# AI 면접 시뮬레이션 프로젝트 종합 보고서

***

## 1. 프로젝트 개요

### 1.1. 기획 배경 및 목적
현대의 채용 플랫폼은 비대면 면접과 서류 평가 절차의 비중이 비약적으로 증가하고 있습니다. 이러한 흐름에 맞추어, 구직자가 실제 면접 환경과 유사한 경험을 사전에 체험하고 스스로 문제점을 파악할 수 있도록 돕는 **AI 기반 면접 시뮬레이션 시스템**의 필요성이 대두되었습니다.

본 프로젝트는 지원자가 이력서를 업로드하면, LLM(대형 언어 모델)이 이력서와 직무 정보를 분석하여 맞춤형 면접 질문을 생성하는 것에서 출발합니다. 이후 텍스트, 음성(STT/TTS), 비전 기술(Wav2Lip을 이용한 립싱크 비디오 생성)을 결합하여 가상의 면접관과 대화하는 듯한 극사실적인 인터랙션을 제공하는 시스템을 구축하는 것을 핵심 목표로 하였습니다.

### 1.2. 주요 서비스 시나리오 시퀀스
1. **사용자 환경**: 구직자(사용자)는 회원가입 및 로그인 후 본인의 이력서(PDF)를 업로드하고 지원 직무를 선택합니다.
2. **맞춤형 면접 시작**: 서버는 PDF에서 텍스트를 추출해 요약하고, Gemini 2.0 Flash 모델을 이용해 첫 번째 질문("자기소개를 부탁드립니다" 등)을 생성한 뒤 이를 TTS(음성 합성)와 Wav2Lip(비디오 생성) 처리하여 브라우저로 전송합니다.
3. **질의 응답 사이클**:
    * 지원자가 웹의 오디오 기능을 통해 답변을 녹음(webm)하여 전송합니다.
    * 서버는 Whisper 또는 Google Web Speech를 통해 지원자의 음성을 텍스트로 변환(STT)합니다.
    * 동시에 지원자의 음성 볼륨(RMS), 무음 비율, 말하기 속도 등을 분석하여 긴장도 및 신뢰도를 측정합니다.
    * LLM은 이전 대화 맥락과 이력서를 고려하여 '직무 기술', '인성 및 가치관' 등의 페이즈(Phase)에 맞추어 꼬리 질문을 동적으로 생성합니다.
4. **결과 분석 및 평가**: 전체 11회 내외의 질문-답변 사이클이 종료되면, 시스템은 각 답변에 대한 개별 평가 기록 및 면접 전체의 총평, 장단점 요약을 백그라운드 태스크로 분석하여 최종 결과 리포트를 제공합니다.

## 2. 기술 스택 (Tech Stack)

이 프로젝트는 안정적인 서비스 운영과 복잡한 AI 모델 처리를 동시에 만족시키기 위해 크게 백엔드, 프론트엔드, AI/ML, 데이터베이스, 컴퓨팅 인프라 환경으로 분리되어 설계되었습니다.

### 2.1. Backend
- **Framework**: `FastAPI (v0.115.8)` - 비동기 처리(Asynchronous)를 기본 지원하여 LLM 및 비디오 생성과 같이 I/O 대기가 긴 작업에 최적화되었습니다.
- **Server**: `Uvicorn (v0.34.0)` - ASGI 서버로 FastAPI 앱을 구동합니다.
- **ORM & DB Access**: `SQLAlchemy (v2.0.38)`, `psycopg2-binary`, `asyncpg`

### 2.2. Frontend
- **HTML/CSS/JS**: 의존성 없는 Vanilla Javascript 구조로 작성하여 백엔드(Jinja2 템플릿 대체 및 정적 서빙)와 매끄럽게 연동되도록 구성하였습니다.
- **Media API**: `MediaRecorder` API를 적극 활용하여 브라우저에서 바로 음성 및 비디오 데이터를 캡쳐합니다.

### 2.3. AI/ML 및 데이터 파이프라인
프로젝트의 성패를 가르는 가장 중요한 요소로 다음과 같은 심층 AI 모델들이 융합되었습니다.
- **LLM 엔진**: `Gemini 2.0 Flash` (via `google-generativeai`) 및 꼬리 질문 문맥 연결을 위한 `LangChain` 활용
- **음성 인식 (STT)**: `OpenAI Whisper (로컬 GPU 기반)` 및 보조 엔진으로 `Google Web Speech API`
- **음성 합성 (TTS)**: `gTTS` 모듈 사용 (안정성 및 라이선스 고려)
- **오디오 분석**: `Librosa`, `SoundFile` 모듈을 통한 주파수 진폭, 무음 분석, VAD 처리
- **비전 / 비디오 생성**: 사용자 경험을 극대화하기 위해 `Wav2Lip` GAN 모델을 활용해 오디오와 이미지(면접관)의 입모양을 실시간 수준으로 동기화 처리
- **문서 Parsing**: `pypdf`, `pymupdf` (PDF 텍스트 추출)

### 2.4. Database & Infrastructure
- **RDBMS**: `PostgreSQL 14+` - 복잡한 외래키 관계(유저, 직무, 진행 상황, 결과 등) 관리
- **Job Queue**: `FastAPI BackgroundTasks` 활용 (향후 Celery/Redis 확장 고려형태)
- **미디어 처리 도구**: `FFmpeg` - H.264 인코딩 및 mp3/wav/webm 형식 간의 상호 변환 파이프라인

## 3. 시스템 아키텍처

본 시스템은 클라이언트의 요청이 무거운 비디오 생성이나 LLM 대기로 인해 블로킹(Blocking)되지 않도록 아키텍처가 설계되었습니다.

### 3.1. 전반적인 시스템 흐름 (Data Flow)
1. **Client Layer**: 브라우저 기반 GUI. 사용자의 이력서(PDF) 업로드 및 MediaStream 객체를 통한 비디오/오디오 스트리밍 송수신.
2. **API Gateway & Routing (FastAPI)**:
    - `/api/auth/`, `/api/user/`: 인증 및 사용자 메타데이터 처리 라우터
    - `/api/interview/`: 이 시스템의 핵심부. 질문 생성, 파일 업로드, STT/TTS 변환 요청 접수
3. **Business Logic & AI Services Layer**:
    - `pdf_service.py`: 업로드된 이력서의 텍스트 파싱.
    - `llm_service.py`: LangChain History와 Gemini를 결합해 이력서 요약, 질문 생성, 답변 평가를 동기/비동기 혼합 워크플로우로 분리 처리.
    - `stt_service.py`: Webm 오디오를 Wav로 변환 후, 노이즈를 필터링(`preprocess_audio`)한 뒤 인식.
    - `video_gen_service.py`: FFmpeg를 시스템 레벨에서 호출하고 GPU 기반의 PyTorch Wav2Lip 모델로 비디오를 랜더링하는 헤비 컴퓨팅 영역.
4. **Data Access Layer**: `psycopg2`를 통한 PostgreSQL 쿼리(Raw Query 및 ORM 혼용).

### 3.2. 비동기 및 백그라운드 태스크 아키텍처
핵심 아키텍처 강점 중 하나는 사용자 대기 시간을 줄이기 위한 **관심사 분리**입니다.
* **동기적 반환 영역**: 클라이언트가 음성을 보내면 즉각적으로 STT를 수행하고, 다음 꼬리 질문(Text)과 그 음성(TTS) 및 비디오(Wav2Lip) 링크만을 신속히 반환합니다.
* **비동기/백그라운드 영역 (`BackgroundTasks`)**: 방금 전 사용자가 답변한 내용에 대한 "평가(Evaluation)" 모델링 작업과 전체 이력서 대비 직무 적합도 평가는 API의 응답 스레드에서 분리되어 `evaluate_answer_bg` 로직으로 백그라운드 환경에서 수행된 후 Database를 갱신합니다.
## 4. 핵심 구현 및 실제 개선 사항

AI 면접 시뮬레이션의 사실감과 기술적 완성도를 높이기 위해 다음과 같은 핵심 기능들이 구현 및 개선되었습니다.

### 4.1. LangChain 기반의 동적 꼬리 질문 생성 로직 (Context Memory)
초기에는 1회성 질문-답변(Stateless) 형태로 LLM API를 호출했으나, 실제 면접처럼 이전 답변에 꼬리를 무는 질문을 생성하기 위해 `LangChain` 프레임워크를 도입하였습니다.
- **구현 방식**: `generate_next_question` 함수에서 지원자의 과거 모든 (질문, 답변) 쌍을 `History(AIMessage, HumanMessage)` 객체 리스트로 구성하여 LLM에 프롬프트로 전달합니다.
- **개선 효과**: 모델이 이전 대화의 맥락(Context)을 완벽히 이해하게 되어, "아까 말씀하신 프로젝트에서 구체적으로 어떤 역할을 하셨나요?"와 같은 자연스러운 심층 질문 생성이 가능해졌습니다.

### 4.2. STT 다중화 및 분석 파이프라인 연동 (`stt_service.py`)
음성 인식의 실패는 서비스 흐름을 완전히 중단시키므로 STT 엔진을 이원화(Fallback)하여 안정성을 극대화했습니다.
- **주력 엔진**: 응답 속도가 빠른 `Google Web Speech API`
- **보조 엔진 (Fallback)**: 로컬 GPU를 활용하는 `OpenAI Whisper (turbo 모델)`
- **오디오 분석**: `Librosa` 라이브러리를 활용해 STT 수행 전/후 과정에서 사용자의 말하기 속도(음절/초), 무음 비율, 볼륨(RMS) 진폭 등을 수치화합니다. 이 데이터는 백그라운드 평가 태스크(`evaluate_answer_bg`)에 전달되어 사용자의 **'긴장도'** 및 **'자신감'**을 유추하는 객관적 지표로 활용됩니다.

### 4.3. 실시간에 가까운 Wav2Lip 비디오 렌더링 (`video_gen_service.py`)
정지된 면접관 이미지(`man.png`)와 TTS로 생성된 오디오를 결합해, 입모양을 정교하게 동기화시키는 Wav2Lip 기술을 통합했습니다.
- **비동기 Subprocess 처리**: 비디오 생성은 GPU를 극도로 소모하는 워크로드이므로 FastAPI 메인 이벤트 루프를 막지 않게 `asyncio.create_subprocess_exec`로 격리 실행합니다.
- **FFmpeg 최적화**: 생성된 비디오가 웹 브라우저에서 즉시 스트리밍(재생)될 수 있도록, 후처리 과정에서 `H.264`, `AAC` 코덱으로 인코딩하여 출력합니다.

## 5. 데이터베이스 설계 (Database Schema)

면접 정보의 생애주기(회원가입 -> 공고 선택 -> 이력서 저장 -> 면접 진행 -> 결과 분석)를 관리하기 위한 스키마 설계입니다.

### 5.1. 주요 테이블 (Tables)
1. **users**: 구직자 탈퇴/수정 처리 및 기본 정보(id_name, pw, email 등)
2. **job_postings**: 회사가 올린 직무 공고 단위 테이블. 여러 종류의 면접을 관리하려면 공고 베이스가 필수적입니다.
3. **interview_information**: 지원자가 특정 직무에 지원할 때 제출한 **이력서(PDF 경로)** 보관
4. **Interview_Progress (핵심)**:
   - 각 질문 턴(Turn)마다 한 행씩 저장됩니다.
   - 구성: `interview_number`(세션 ID), `Create_Question`(질문), `Question_answer`(답변 STT), `Answer_Evaluation`(답변 개별 평가 정보), `audio_path`(오디오 보관 경로)
5. **Interview_Result**: 한 `interview_number` 세션이 END에 도달하면 백그라운드 태스크로 구동된 종합 분석 결과(합격/불합격 예측, 장단점, 피드백)가 적재되는 테이블입니다.

### 5.2. 설계상 고려사항
면접 도중 사용자의 인터넷이 끊기거나 새로고침 되더라도 "마지막으로 진행한 질문"을 `Interview_Progress` 테이블의 가장 최신 ID(`ORDER BY id DESC LIMIT 1`)로 추적하여 면접을 즉시 복구/재개할 수 있도록 설계되었습니다.

## 6. 실제 운영 중 발생한 문제 및 해결 방안 (Troubleshooting)

작업 일지(docx)와 소스코드 리뷰를 통해 도출된 대표적인 트러블슈팅 사례입니다.

### 6.1. FFmpeg 인코딩 대기 시간 및 경로 문제
- **문제상황**: 브라우저에서 녹음된 webm 오디오를 곧바로 Wav2Lip 모델에 넘겼으나 차원/프레임 레이트 문제로 렌더링에 실패하거나, 최종 결과물 mp4가 웹에서 재생되지 않는(코덱 미지원) 현상 발생.
- **해결과정**: 비디오 생성 파이프라인의 전/후단에 FFmpeg 컨버팅 로직을 강제 삽입.
  - 전단: 입력 오디오를 `16kHz, mono, pcm_s16le` 규격의 wav로 평탄화.
  - 후단: 결과 영상을 `libx264, yuv420p` 픽셀 포맷으로 재인코딩.
- **결과**: 브라우저 호환성 문제 100% 해결.

### 6.2. LLM 환각(Hallucination) 및 출력 포맷 붕괴
- **문제상황**: Gemini 모델에 JSON 형태의 반환을 요구했으나, 간헐적으로 마크다운 코드블록(```json ... ```)을 포함하여 반환해 파싱 에러(`JSONDecodeError`) 유발.
- **해결과정**: `clean_json_string` 유틸리티 함수를 서비스 레이어 전체에 도입하여 코드블록 정규식을 선 제거 후 파싱.
- **결과**: JSON API의 파이프라인 안정성이 크게 향상되었습니다.

### 6.3. 백그라운드 평가 태스크(Background Tasks) 분리로 응답성 향상
- **문제상황**: 사용자가 답변을 완료하면, STT -> 답변 평가(LLM) -> 다음 질문 생성(LLM) -> TTS 변환 -> 동영상 렌더링 까지 동기적으로 수행되어 약 20~30초의 로딩(지연)이 발생.
- **해결과정**: **"다음 질문 생성"**과 **"작성한 답변에 대한 개별 평가"**를 분리 구조로 변경. 개별 평가는 응답 시점을 블로킹하지 않고 FastAPI의 `BackgroundTasks`(`evaluate_answer_bg`)로 던져서 백그라운드에서 진행 및 DB 업데이트를 하도록 수정.
- **결과**: 사용자 체감 대기 시간 약 30% 이상 단축.

### 6.4. 질문 제한 시간 초과 후 답변 비활성화 오류
- **문제상황**: 면접 진행 중 질문 제한 시간(90초)이 초과되어 자동으로 다음 질문으로 넘어갈 때, "답변 시작" 버튼이 비활성화된 상태로 유지되어 사용자가 다음 질문에 답변할 수 없는 치명적인 버그가 발생.
- **해결과정**: 프론트엔드(`interview.js`)의 `forceTimeOutSubmit` 함수가 실행될 때, `mediaRecorder`(녹음기)와 타이머 상태가 다음 질문으로 넘어갈 때까지 완전히 초기화되지 않는 문제를 식별. 타이머 인터벌을 즉시 초기화(`clearInterval`) 및 강제 정지하고, 새로운 질문이 로드될 때 녹음기가 활성화되어 있다면 강제 종료(`stop()`) 시킨 후 버튼 상태를 복구하도록 수정.
- **결과**: 시간 초과로 자동으로 질문이 넘어가더라도 다음 질문에서 정상적으로 답변을 시작할 수 있게 됨. 시간 초과 시에도 부분 녹음 또는 "대답하지 않음" 처리 로직이 보강됨.

### 6.5. 음성 답변 결과 파싱 오류 (`undefined` 반환 이슈)
- **문제상황**: 브라우저에서 제출된 음성 답변의 결과 텍스트와 개별 평가 점수가 `undefined`로 표시되는 문제 발생.
- **해결과정**: 클라이언트(`interview.js`)는 서버 응답에서 `stt` 필드와 `score` 필드를 기대했으나, 백엔드 API `/submit`의 반환 명세에 해당 필드가 누락되어 있었음을 확인. `api/interview.py`를 수정하여 최종 반환될 `JSON` 객체에 `text_content`를 `stt` 필드로, `evaluation.get("score")` 값을 `score` 필드로 매핑하여 전달 
- **결과**: 제출한 음성 답변이 텍스트 형태와 최종 평가 점수(숫자)로 화면에 정상 표기됨.

### 6.6. STT 클래스 메서드 호출 간 `AttributeError`
- **문제상황**: 음성 답변 시 `(Audio transcription unavailable)` 메시지가 출력되며 답변이 인식되지 않고 면접 단계가 꼬이는 문제 발생.
- **해결과정**: 서버 API(`api/interview.py`)가 모듈 레벨의 함수가 아닌 존재하지 않는 `stt_service.transcribe` 를 호출하고 있었음. (실제로는 `transcribe_audio` 거나 클래스 메서드임) `except` 블록에 포획되면서 원인이 수면 위로 구체화되지 않고 에러 메시지 텍스트 렌더링에 그쳤던 것.
- **결과**: 의존성 주입 시 메서드 호출부를 명확히 `stt_service.transcribe_audio(file_path)` 로 변경하여 음성 추출 프로세스를 정상화.

### 6.7. 면접 질문 미출력 및 직무 컨텍스트 누락 문제
- **문제상황**: 1~10단계로 구성된 면접 프로세스에서 질문이 정상적으로 생성되거나 출력되지 않고 멈추는 현상 발생.
- **해결과정**: DB(`interview.db`)에서 질문을 불러올 때 직무 조건 검토가 불완전했고, 단계별(1: 자기소개, 2~4: 인성, 5~9: 직무, 10: 마무리) 프로세스가 정립되지 않은 문제를 식별. 클라이언트가 질문을 별도 전송하지 않더라도 백엔드의 세션(Memory/DB)에 `current_question`을 캐싱하여 LLM이 이를 인지하도록 `interview_service.py`를 대대적으로 리팩토링.
- **결과**: 1단계부터 10단계까지의 정해진 플로우와 추가 꼬리질문이 혼선 없이 자연스럽게 생성·실행됨.

### 6.8. Pyodide 브라우저 환경에서의 JsProxy 객체 인덱싱 에러
- **문제상황**: 프론트엔드 환경에 도입된 로직에서 `Uncaught PythonError: TypeError: 'pyodide.ffi.JsProxy' object is not subscriptable` 에러 발생. 음성 인식 이벤트의 결과 배열 파싱 도중 프로그램이 중지됨.
- **해결과정**: `event.results[i][0].transcript` 처럼 중첩 리스트 문법(`[]`)으로 브라우저(JsProxy)의 반환 객체에 접근하려던 코드가 Pyodide 제약에 걸린 것임을 파악. 자바스크립트 DOM API의 표준 방식인 `.item(i)` 메서드를 이용해 `event.results.item(i).item(0).transcript` 방식으로 치환.
- **결과**: 음성 인식 결과를 프론트에 정상적으로 반영하고 치명적 에러 해결.

## 7. 기술적 특징 요약

이 시스템을 관통하는 3대 기술적 특징은 다음과 같이 요약할 수 있습니다.

1. **초연결성 (Hyper-connectivity)**
   단순 문답형 챗봇에서 벗어나, 이력서(텍스트) -> LLM 생성(텍스트) -> TTS(음성) -> Wav2Lip(비전) -> 지원자 답변(음성) -> STT 및 오디오 분석(텍스트 및 수치)으로 이어지는 **멀티모달리티(Multimodality)** 파이프라인이 끊김 없이 연결되어 있습니다.

2. **비동기 기반의 논블로킹 최적화 (Non-blocking I/O)**
   FastAPI의 특성을 십분 활용하여, 무거운 AI 연산(Whisper, FFmpeg 렌더링, LLM 답변 평가)을 백그라운드 태스크나 별도의 프로세스(Subprocess Exec)로 밀어내고 클라이언트(브라우저)에게는 즉시 스트리밍 이벤트를 반환해 대기 시간을 숨겼습니다.

3. **장애 허용 아키텍처 (Fault Tolerance)**
   STT 로직에서 메인 엔진(Google API)이 무응답일 경우 즉시 로컬 GPU 엔진(Whisper)으로 전환하는 다중화(Fallback) 구조를 채택하여, 핵심 도메인인 '면접 진행'이 중도에 중단되는 것을 원천 차단했습니다.

## 8. 프로젝트 차별점

기존 시장의 상용 AI 면접 서비스(뷰아이, 인페이스 등) 또는 토이 프로젝트들과 비교했을 때 다음과 같은 독보적인 우위점이 존재합니다.

- **맥락 인지형(꼬리물기) 질문 생성**: 고정된 질문 은행 방식이 아닌, `LangChain`의 메모리 컨텍스트를 활용하여 방금 지원자가 한 답변의 허점을 파고드는 심층 압박 면접 구성을 실시간으로 구현해 냈습니다.
- **오디오 비언어적 큐(Cue) 분석 도입**: 지원자가 말하는 텍스트(Transcript)의 내용뿐만 아니라, 망설임(무음 시간 비중), 말의 빠르기 변동, 진폭(RMS) 척도를 종합하여 긴장도 및 자신감 수치로 역산해 내는 다차원적 지원자 평가 기능을 자체 구축했습니다.
- **오픈소스 기반 딥페이크 립싱크 활용**: 고비용의 클라우드 Avatar API에 의존하지 않고, 직접 내부 서버에서 `Wav2Lip` GAN 모델을 커스터마이징 및 경량화 서빙하여 면접관 영상을 실시간으로 생성합니다. 이는 운영 비용을 획기적으로 낮추는 요소입니다.

## 9. 프로젝트 회고 (Lessons Learned)

개발부터 운영, 트러블슈팅에 이르는 전 과정을 통해 얻은 기술적/비즈니스적 인사이트입니다.

*   **비동기 프로그래밍 모델의 중요성 체감**: 수 초 단위의 지연이 발생하는 무거운 딥러닝 추론 작업(Video Generation)을 웹 API 메인 스레드에서 분리하는 것이 서버 아키텍처에 얼마나 큰 영향을 미치는지 깨달았습니다.
*   **Prompt Engineering의 한계와 극복**: 생성형 모델(Gemini)에 단순히 프롬프트를 주는 것만으로는 완벽한 JSON 포맷을 고정하기 어렵다는 점(환각 문제)을 겪었으며, 결국 애플리케이션 레벨의 방어 로직(`clean_json_string`)이 반드시 수반되어야 함을 배웠습니다.
*   **Media 데이터 규격의 늪 (FFmpeg)**: 브라우저가 생성하는 webm, 모델이 요구하는 wav, 최종 재생을 위한 H.264 mp4 간의 인코딩 규격 차이에서 오는 블랙박스 오류들을 디버깅하면서 코덱과 컨테이너 구조에 대한 깊은 이해를 확보했습니다.

## 10. 확장 계획 (Future Work & Scalability)

본 시스템은 현재의 안정화 단계를 넘어, 실제 대규모 서비스로 도약하기 위해 다음 3가지 확장 로드맵을 지향하고 있습니다.

1. **분산 추론 서버 구축 (Nvidia Triton / Ray Serve)**: 현재 단일 서버 내에서 구동되는 Wav2Lip과 Whisper 모델을 추론 전용 프레임워크인 `Triton Inference Server`로 이관하여 GPU 연산의 병렬 처리량(Throughput)을 고도화할 계획입니다.
2. **Message Queue (Celery + Redis) 전면 도입**: `BackgroundTasks`로 처리 중인 딥러닝 워크로드와 최종 결과 모델링 작업을 완벽한 스케일 아웃이 가능한 분산 비동기 큐 시스템 기반 워커(Worker) 아키텍처로 분리합니다.
3. **면접관 감정/태도 커스터마이징 (SadTalker 통합)**: 현재 스탠다드 버전으로 사용 중인 정지 이미지 영상화를 넘어, 압박 면접 시에는 날카로운 표정, 편안한 면접 시에는 웃는 표정이 생성되도록 `LivePortrait` 혹은 `SadTalker` 파이프라인의 감정 벡터를 동적 제어하는 기능을 추가로 연동할 예정입니다.
