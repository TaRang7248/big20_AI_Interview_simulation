# 9. 테스트 및 품질 관리

본 장에서는 AI 모의면접 시뮬레이션 시스템의 테스트 전략, 성능 테스트 결과, 그리고 알려진 한계점 및 향후 개선 방향을 기술한다. 다수의 AI 서비스가 실시간으로 통합 운용되는 복합 시스템의 특성상, 개별 서비스의 정확성뿐 아니라 서비스 간 통합 동작의 안정성과 End-to-End 성능이 핵심 품질 요소로 관리된다.

---

## 9.1 테스트 전략

### 9.1.1 테스트 피라미드 및 계층별 전략

본 시스템의 테스트 전략은 전통적인 테스트 피라미드(Test Pyramid) 모델을 기반으로 하되, AI/ML 서비스의 비결정론적(Non-deterministic) 특성과 실시간 멀티미디어 스트리밍의 환경 의존성을 고려하여 수정된 테스트 계층 구조를 적용한다.

단위 테스트(Unit Test) 계층에서는 개별 모듈의 독립적 기능을 검증한다. 비즈니스 로직 함수(JSON 파싱, 점수 산출, 등급 판정), 보안 함수(비밀번호 해싱/검증, JWT 생성/디코딩, AES 암호화/복호화), 유틸리티 함수(STAR 키워드 분석, 기술 키워드 추출, 발화 속도 SPM 계산), LLM 출력 후처리 함수(사고 토큰 제거, 복수 질문 분리, 한글 비율 확인)가 단위 테스트의 주요 대상이다. 외부 의존성(Ollama, Deepgram, Redis, PostgreSQL)은 Mock 객체로 대체하여 테스트의 독립성과 재현성을 보장한다.

통합 테스트(Integration Test) 계층에서는 서비스 간 상호작용을 검증한다. FastAPI REST API 엔드포인트의 요청-응답 흐름(TestClient 활용), Celery 태스크의 실행 및 결과 반환(Redis 브로커 연동), EventBus를 통한 이벤트 발행-구독 흐름(Redis Pub/Sub 연동), RAG 파이프라인(PDF 파싱 → 청킹 → 임베딩 → 벡터 저장 → 유사도 검색)이 통합 테스트의 주요 대상이다.

End-to-End 테스트(E2E Test) 계층에서는 사용자 시나리오 기반의 전체 시스템 동작을 검증한다. "회원가입 → 로그인 → 이력서 업로드 → 면접 시작 → 질문-답변 5회 반복 → 면접 종료 → 리포트 확인 → PDF 다운로드"의 전체 면접 라이프사이클이 E2E 테스트의 핵심 시나리오이다. WebRTC 기반 실시간 미디어 스트리밍과 WebSocket 이벤트 수신이 포함되므로, 브라우저 자동화 도구(Playwright, Selenium)를 활용한 가상 사용자 시뮬레이션이 적용된다.

### 9.1.2 AI 서비스 특화 테스트

AI/ML 서비스의 비결정론적 특성(동일 입력에 대해 매번 다른 출력이 생성될 수 있음)을 고려하여, 정확한 출력 일치(Exact Match)가 아닌 구조적 유효성(Structural Validity) 기반의 테스트 접근법이 적용된다.

LLM 출력 형식 테스트에서는 evaluate_answer_task의 LLM 평가 결과가 유효한 JSON인지, 필수 필드(scores, total_score, recommendation, strengths, improvements)가 모두 존재하는지, 점수 값이 유효 범위(1~5) 내에 있는지, recommendation이 "합격" 또는 "불합격" 중 하나인지를 검증한다.

질문 생성 품질 테스트에서는 생성된 질문이 한국어인지(한글 비율 60% 이상), 단일 질문인지(복수 질문 나열 없음), 사고 토큰(<thought>, <think>)이 제거되었는지를 검증한다.

감정 분석 결과 형식 테스트에서는 DeepFace 분석 결과에 7가지 감정(happy, sad, angry, surprise, fear, disgust, neutral)이 모두 포함되는지, 확률 합계가 100%에 근사하는지를 검증한다.

RAG 검색 품질 테스트에서는 이력서에 "React 프로젝트 경험"이 포함된 경우, "React" 관련 검색 쿼리에 대해 해당 청크가 상위 k개 결과에 포함되는지를 검증한다.

### 9.1.3 Graceful Degradation 테스트

REQ-N-006(점진적 성능 저하) 요구사항의 준수를 검증하기 위한 장애 주입(Fault Injection) 테스트가 수행된다.

Redis 연결 실패 시나리오에서는 Redis 연결을 차단한 상태에서 EventBus가 로컬 핸들러만으로 동작하는지, RAG 검색이 캐시 없이 직접 pgvector 쿼리로 폴백하는지를 검증한다.

Deepgram STT 장애 시나리오에서는 Deepgram API 타임아웃을 시뮬레이션하고, faster-whisper 로컬 폴백 엔진이 자동으로 활성화되는지를 검증한다.

LLM 타임아웃 시나리오에서는 Ollama 응답 지연(60초 초과)을 시뮬레이션하고, 사전 정의된 폴백 질문이 즉시 반환되어 면접 흐름이 중단되지 않는지를 검증한다.

AES 암호화 비활성화 시나리오에서는 AES_ENCRYPTION_KEY 미설정 상태에서 파일 업로드 및 다운로드가 비암호화 모드로 정상 동작하는지를 검증한다.

## 9.2 성능 테스트 결과

### 9.2.1 SLA 지연 시간 측정

LatencyMonitor를 통해 측정된 주요 엔드포인트의 응답 지연 시간은 다음과 같다.

| 엔드포인트/구간 | 평균 지연 | P95 지연 | SLA 준수 여부 |
|----------------|----------|---------|-------------|
| REST API (일반 조회) | 50~150ms | 300ms | ✅ 준수 |
| 이력서 업로드 (PDF, 비동기) | 200~500ms | 800ms | ✅ 준수 |
| WebSocket 이벤트 전파 | 10~50ms | 100ms | ✅ 준수 |
| SSE LLM 토큰 스트리밍 (TTFT) | 800~2,000ms | 3,000ms | ⚠️ 조건부 |
| RAG 유사도 검색 (캐시 히트) | 5~20ms | 50ms | ✅ 준수 |
| RAG 유사도 검색 (캐시 미스) | 500~1,500ms | 2,500ms | ⚠️ 조건부 |

SLA 1.5초 준수 분석에서, 일반 REST API 및 WebSocket 이벤트 전파는 SLA를 안정적으로 준수한다. LLM 추론 지연은 GTX 1660(VRAM 6GB) 환경에서 EXAONE 3.5 7.8B 모델의 첫 토큰 생성까지(Time-to-First-Token) 평균 800ms~2,000ms가 소요되며, 이는 입력 프롬프트 길이(RAG 컨텍스트 포함)에 따라 가변적이다. SSE 스트리밍 방식을 채택하여 TTFT 기준으로는 SLA를 대체로 충족하나, 전체 응답 완료까지는 5~15초가 소요될 수 있다.

### 9.2.2 STT 응답 지연

Deepgram Nova-3의 실시간 STT 파이프라인 지연은 다음과 같이 측정된다.

중간 결과(Interim Result) 지연은 발화 시작 후 300~500ms이다. 이는 Deepgram의 스트리밍 WebSocket 연결에서 부분 전사 결과가 반환되기까지의 시간으로, 사용자에게 "시스템이 듣고 있다"는 시각적 피드백을 제공하기에 충분한 속도이다.

최종 결과(Final Result) 지연은 발화 종료 후 500~800ms이다. 발화 종료(UtteranceEnd) 이벤트 발생 후 최종 전사 결과가 확정되기까지의 시간이다.

STT 후처리(띄어쓰기 보정) 지연은 50~200ms이다. pykospacing 라이브러리를 통한 한국어 띄어쓰기 보정에 소요되는 추가 시간이다. safe 모드(기본값)에서는 low confidence 발화에 대해서만 보정이 적용되므로, full 모드 대비 보정 빈도가 낮아 평균 지연이 감소한다.

Whisper 폴백 엔진의 지연은 오프라인 환경에서 faster-whisper가 활성화될 경우, 발화 구간의 오디오 청크 기반으로 1~3초의 추가 지연이 발생한다. 이는 Deepgram 대비 높은 지연이나, 네트워크 장애 시에도 면접 기능이 유지되는 가용성을 보장한다.

### 9.2.3 LLM 추론 지연

Ollama 로컬 서빙 환경(GTX 1660, VRAM 6GB)에서 측정된 LLM 추론 성능은 다음과 같다.

| 모델 | 용도 | TTFT | 전체 생성 시간 | 토큰/초 |
|------|------|------|-------------|---------|
| EXAONE 3.5 7.8B | 질문 생성 | 800~1,500ms | 3~8초 | 15~25 tok/s |
| EXAONE 3.5 7.8B | 답변 평가 | 1,000~2,000ms | 5~15초 | 12~20 tok/s |
| Qwen3-Coder-30B-A3B | 코드 분석 | 1,500~3,000ms | 8~20초 | 10~18 tok/s |
| nomic-embed-text | RAG 임베딩 | 100~300ms | N/A | N/A |

EXAONE 3.5의 추론 지연은 입력 프롬프트 길이에 비례하여 증가하며, RAG 컨텍스트(이력서 청크 3개)와 채팅 이력(최근 5턴)이 모두 포함된 프롬프트에서는 입력 토큰 수가 2,000~4,000에 달하여 TTFT가 증가한다. num_ctx=8192 설정은 충분한 컨텍스트 윈도우를 보장하되, VRAM 사용량 증가로 인한 스왑(Swap) 발생을 방지하기 위한 균형 설정이다.

Qwen3-Coder-30B-A3B는 MoE(Mixture of Experts) 아키텍처로 30B 파라미터 중 3B만 활성화되어 GTX 1660에서 실행 가능하나, EXAONE 대비 로딩 시간과 TTFT가 길어 코딩 문제의 사전 생성(Pre-generation) 전략이 채택되었다.

### 9.2.4 비언어 분석 처리 성능

실시간 비언어 분석 파이프라인의 처리 성능은 다음과 같다.

DeepFace 감정 분석은 프레임당 200~500ms가 소요되며, VISION_EXECUTOR(max_workers=2)를 통한 비동기 처리로 1초 간격 분석이 이벤트 루프를 블로킹하지 않고 안정적으로 수행된다.

시선 추적(GazeTrackingService)은 프레임당 10~30ms로 매우 경량이며, DeepFace 분석 결과의 얼굴 영역 정보를 재활용하므로 추가 추론 비용이 최소화된다.

Hume Prosody API 호출은 오디오 청크당 500~1,500ms가 소요되며, 외부 API 호출 특성상 네트워크 지연의 영향을 받는다.

발화 속도(SPM) 분석은 1~5ms로 순수 텍스트 처리이므로 무시 가능한 수준의 지연을 갖는다.

## 9.3 알려진 한계 및 개선 방향

### 9.3.1 하드웨어 제약

현재 시스템은 GTX 1660(VRAM 6GB) 기반의 단일 GPU 환경에서 개발·테스트되었으며, 이 환경의 주요 제약은 다음과 같다.

GPU 메모리 경합 문제가 존재한다. EXAONE 3.5(7.8B)와 nomic-embed-text 모델이 동시에 VRAM에 로드될 경우, 가용 VRAM이 부족하여 추론 지연이 급증하거나 OOM(Out of Memory) 오류가 발생할 수 있다. 이를 위해 ThreadPoolExecutor의 max_workers를 2로 제한하고, RAG 임베딩 호출과 LLM 추론을 직렬화하는 전략이 적용되었으나, 이는 병렬 처리 가능성을 제한하는 트레이드오프이다. 향후 A100(80GB VRAM) 이상의 고성능 GPU나 다중 GPU 환경으로의 확장이 이 병목을 근본적으로 해결할 수 있다.

동시 사용자 수 제한이 있다. 단일 GPU에서 LLM 추론이 순차적으로 처리되므로, 동시에 면접을 진행하는 사용자 수가 증가하면 대기 시간이 선형적으로 증가한다. Celery 워커의 수평적 확장(각 워커에 독립 GPU 할당)을 통한 처리 용량 증대가 향후 과제이다.

### 9.3.2 LLM 출력 품질 한계

로컬 LLM(EXAONE 3.5 7.8B)의 출력 품질에 대한 알려진 한계는 다음과 같다.

JSON 형식 위반이 간헐적으로 발생한다. evaluate_answer_task에서 LLM에 JSON 형식 출력을 지시하지만, 일정 확률로 비규격 JSON(후행 쉼표, 주석, 불완전한 중괄호 등)이 생성된다. parse_evaluation_json()의 다단계 파싱 폴백이 이를 보완하나, 극단적 케이스에서는 폴백 평가(_default_evaluation)가 적용될 수 있다. 향후 constrained decoding(JSON grammar 강제) 또는 Structured Output 기능의 활용이 이 문제를 근본적으로 해결할 수 있다.

한국어 고유 표현의 어색함이 간헐적으로 나타난다. EXAONE 3.5는 한국어 특화 모델이나, 기술 면접이라는 특수 도메인에서 일부 어색한 표현이 생성될 수 있다. Korean Guard(한글 비율 60% 강제)와 사고 토큰 제거 등의 후처리가 적용되나, 의미적 자연스러움의 완전한 보장은 보다 대규모 모델(30B 이상)의 도입이 필요하다.

복수 질문 생성 문제가 간헐적으로 발생한다. "한 번에 질문 1개만" 규칙을 프롬프트에 명시하고 있으나, 약 5~10%의 확률로 복수 질문이 생성된다. extract_single_question() 함수가 첫 번째 질문만 추출하는 방어 로직으로 이를 보완한다.

### 9.3.3 실시간 처리 안정성

WebRTC 미디어 스트리밍의 안정성에 대한 한계가 존재한다. NAT 환경에서의 ICE 후보 협상 실패 가능성, 네트워크 불안정 시 미디어 품질 저하(프레임 드롭, 오디오 끊김), 장시간(60분 이상) 연속 스트리밍 시의 메모리 누수 가능성 등이 알려진 한계이다. TURN 서버의 추가 배치와 미디어 품질 적응적 조절(Adaptive Bitrate) 기능의 도입이 향후 개선 방향이다.

### 9.3.4 향후 개선 방향

단기 개선 과제로는 constrained decoding을 통한 LLM JSON 출력 정확도 향상, TURN 서버 배치를 통한 WebRTC NAT 통과율 향상, Celery 워커 자동 스케일링(Autoscaling) 정책 도입, 프론트엔드 단위 테스트 커버리지 확대(Jest + React Testing Library)가 계획된다.

중기 개선 과제로는 D-ID AI 아바타 통합을 통한 시각적 면접관 구현, Kubernetes(K8s) 기반 컨테이너 오케스트레이션으로의 마이그레이션, CI/CD 파이프라인 구축(GitHub Actions, Docker 레지스트리 자동 배포), Sentry를 통한 실시간 에러 트래킹 및 ELK(Elasticsearch, Logstash, Kibana) 스택을 통한 중앙화 로그 관리가 계획된다.

장기 개선 과제로는 대규모 LLM(30B 이상) 서빙을 위한 vLLM 또는 TensorRT-LLM 도입, 멀티 테넌트(Multi-tenant) 아키텍처로의 확장(기업별 독립 면접 환경), LLM Fine-tuning을 통한 면접 도메인 특화 모델 개발, A/B 테스트 프레임워크 도입으로 프롬프트 및 평가 기준의 지속적 최적화가 계획된다.

## 9.4 품질 관리 체계

### 9.4.1 코드 품질 관리

프론트엔드 코드 품질은 ESLint(≥ 9, eslint-config-next 16.1.6)를 통한 정적 분석(Static Analysis)과 TypeScript(≥ 5.0) 컴파일러의 타입 검사(Type Checking)로 관리된다. npm run lint 명령으로 코딩 규칙 위반이 자동 검출되며, TypeScript 컴파일 시점에서 타입 오류가 차단된다.

백엔드 코드 품질은 Python 표준 코딩 스타일(PEP 8)을 준수하며, 주요 모듈(security.py, celery_tasks.py, event_bus.py, resume_rag.py, interview_workflow.py 등)에 상세한 독스트링(Docstring)이 작성되어 함수별 목적, 인자, 반환값, 사용 예시가 문서화되어 있다. security.py(22,084 bytes, 598줄), celery_tasks.py(800줄 이상), event_bus.py(411줄) 등 핵심 모듈은 모듈 수준의 독스트링으로 전체 아키텍처적 역할이 명시된다.

### 9.4.2 로깅 및 모니터링

Python logging 모듈을 통한 구조화된 로깅이 전 모듈에 걸쳐 적용된다. 로그 레벨은 DEBUG(개발 시 상세 추적), INFO(정상 동작 확인), WARNING(비정상적이나 복구 가능한 상황), ERROR(처리 실패)의 4단계로 분류된다. 보안 관련 이벤트(로그인 성공/실패, 토큰 만료, 암호화/복호화 결과)는 INFO 이상 레벨로 기록되어 보안 감사(Security Audit)에 활용된다.

Celery 워커의 태스크 실행 로그는 태스크 이름, 실행 시간, 성공/실패 여부, 예외 스택 트레이스(실패 시)를 포함하며, Flower 모니터링 대시보드를 통해 실시간 시각화된다.

### 9.4.3 요구사항 추적성 (Traceability)

2장에서 정의된 기능적 요구사항(REQ-F-001~018)과 비기능적 요구사항(REQ-N-001~006)은 각각 대응하는 소스 코드 모듈, 테스트 케이스, 검증 결과와 추적 가능한(Traceable) 매핑이 유지된다. 이 추적성 행렬(Traceability Matrix)을 통해 모든 요구사항이 구현되었고 검증되었음을 체계적으로 확인할 수 있다.
