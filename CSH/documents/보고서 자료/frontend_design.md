# 6. 프론트엔드 설계 및 구현

본 장에서는 AI 모의면접 시뮬레이션 시스템의 프론트엔드 계층(Presentation Layer)의 설계 원칙, 페이지 구성, 컴포넌트 아키텍처, 상태 관리 전략, 실시간 이벤트 처리 시스템, 반응형 설계 및 데이터 시각화 구현을 체계적으로 기술한다. 프론트엔드는 Next.js 16.1.6(App Router), React 19.2.3, TypeScript, Tailwind CSS v4의 기술 스택으로 구현되었으며, 지원자와 AI 면접관 간의 실시간 인터랙션을 가능케 하는 웹 기반 사용자 인터페이스를 제공한다.

---

## 6.1 페이지 구성 (10개 페이지)

본 시스템의 프론트엔드는 Next.js App Router의 파일 기반 라우팅(File-based Routing)을 활용하여 10개의 주요 페이지(Route)로 구성된다. 각 페이지는 src/app/ 디렉토리 내의 폴더 구조에 대응하며, page.tsx 파일이 해당 라우트의 렌더링을 담당한다.

### 6.1.1 메인 랜딩 페이지 (/)

루트 경로(/)에 위치하는 시스템의 첫 진입 페이지이다. AI 모의면접 시스템의 소개, 주요 기능 안내(화상 면접, 코딩 테스트, 아키텍처 설계), 로그인/회원가입 진입점을 제공한다. HTML lang 속성은 "ko"로 설정되어 한국어 웹 접근성 기준을 충족하며, SEO 메타데이터(title: "AI 모의면접 시스템", description: "AI 기반 모의면접 시뮬레이션 — 화상 면접, 코딩 테스트, 아키텍처 설계")가 Next.js Metadata API를 통해 서버 측에서 생성된다.

### 6.1.2 대시보드 페이지 (/dashboard)

로그인 후 사용자의 주요 허브 역할을 하는 대시보드 페이지이다. 진행 중인 면접 세션, 완료된 면접 결과 목록, 이력서 업로드 상태, 채용공고 목록에 대한 개괄적 뷰를 제공한다. 이력서 업로드 시 Celery의 process_resume_task가 비동기적으로 처리되며, EventBus를 통해 처리 완료 이벤트가 수신되면 Toast 알림으로 사용자에게 실시간 알림이 전달된다.

### 6.1.3 면접 세션 페이지 (/interview)

AI 화상 면접이 실제로 진행되는 핵심 페이지이다. WebRTC를 통한 실시간 비디오/오디오 스트리밍, AI 면접관의 질문 텍스트 SSE 스트리밍 표시, 실시간 감정 분석 결과 인디케이터, STT 중간/최종 변환 결과 표시, VAD 상태 표시, 채팅 이력(ChatBubble 컴포넌트) 등 다수의 실시간 UI 요소가 통합된 복합 페이지이다. WebSocket 연결을 통해 서버에서 발행되는 다양한 이벤트(stt.result, emotion.analyzed, question.generated 등)를 수신하여 각 UI 컴포넌트의 상태를 실시간으로 갱신한다.

### 6.1.4 코딩 테스트 페이지 (/coding)

Monaco Editor(@monaco-editor/react)를 통합한 웹 기반 코드 편집 환경을 제공하는 페이지이다. 5개 프로그래밍 언어(Python, JavaScript, Java, C, C++)의 구문 강조(Syntax Highlighting)를 지원하며, 언어 선택 드롭다운, 코드 편집 영역, 문제 설명 패널, 실행 결과/테스트 결과 패널, AI 코드 분석 피드백 표시 영역으로 구성된다. 코드 제출 시 Docker 샌드박스 실행 결과와 Qwen3-Coder AI 분석 결과가 순차적으로 반환되어 표시된다.

### 6.1.5 화이트보드 페이지 (/whiteboard)

HTML5 Canvas API를 활용한 시스템 설계 화이트보드 인터페이스를 제공하는 페이지이다. 드로잉 도구(도형, 화살표, 텍스트 등), 색상 선택, 실행 취소/재실행 기능이 제공되며, Canvas 렌더링 데이터를 PNG 이미지로 캡처하여 Claude 3.5 Sonnet Vision API에 전송하는 기능이 구현된다. AI 아키텍처 평가 결과(5개 차원 점수 및 피드백)가 사이드 패널에 표시된다.

### 6.1.6 감정 분석 페이지 (/emotion)

실시간 감정 분석 결과를 전용 뷰로 시각화하는 페이지이다. DeepFace 기반 7가지 감정의 실시간 확률 분포를 바 차트로 표시하며, 시선 추적 상태(정면 응시/이탈)와 현재 지배적 감정(Dominant Emotion)을 실시간으로 갱신하여 보여준다.

### 6.1.7 채용공고 관리 페이지 (/jobs)

채용담당자가 채용공고를 생성·조회·수정·삭제(CRUD)할 수 있는 관리 페이지이다. 채용공고에는 회사명, 직무명, 직무 설명, 요구 기술 스택, 경력 요건 등의 필드가 포함되며, 지원자는 채용공고를 선택하여 해당 직무에 특화된 면접을 시작할 수 있다.

### 6.1.8 채용담당자 페이지 (/recruiter)

채용담당자 전용 대시보드로, 지원자의 면접 결과 열람, 합격/불합격 통계, 지원자별 평가 리포트 조회 기능을 제공한다.

### 6.1.9 프로필 페이지 (/profile)

사용자 개인 정보 관리 페이지이다. 프로필 정보 수정, 비밀번호 변경, 소셜 로그인 연동 관리, 이력서 관리(업로드/삭제/RAG 재처리) 기능이 제공된다.

### 6.1.10 설정 페이지 (/settings)

시스템 설정 페이지로, 면접 관련 설정(최대 질문 수, 언어 설정), 알림 설정(Toast 알림 활성화/비활성화), 개인정보 관리(GDPR 전체 데이터 삭제 요청) 등의 사용자 설정을 관리한다.

## 6.2 컴포넌트 아키텍처 (공통/인증/감정/리포트)

프론트엔드 컴포넌트는 src/components/ 디렉토리에 기능적 역할에 따라 4개의 하위 그룹으로 분류되어 관리된다. 각 컴포넌트는 단일 책임 원칙(Single Responsibility Principle)에 따라 설계되어, 하나의 컴포넌트가 하나의 명확한 UI 책임만을 담당하며, 컴포넌트 간의 데이터 전달은 Props 또는 Context를 통해 이루어진다.

### 6.2.1 공통 컴포넌트 (components/common/)

공통 컴포넌트는 시스템 전체에서 재사용되는 범용 UI 요소를 제공한다.

Header.tsx(10,628 bytes)는 시스템의 전역 네비게이션 헤더를 구현한다. 로그인 상태에 따른 조건부 메뉴 렌더링(비로그인: 로그인/회원가입 버튼, 로그인: 대시보드/면접/프로필 메뉴 및 로그아웃 버튼), 로고, 반응형 모바일 햄버거 메뉴가 포함된다. AuthContext를 구독하여 인증 상태 변화에 즉각 반응한다.

Modal.tsx(5,522 bytes)는 재사용 가능한 모달(Modal) 오버레이 컴포넌트를 구현한다. 배경 딤(Dim) 처리, ESC 키/외부 클릭에 의한 닫기, Portal 렌더링을 통한 z-index 관리, 진입/퇴장 애니메이션이 포함된다. LoginModal, RegisterModal, ForgotPasswordModal 등 다수의 인증 관련 모달의 기반 컴포넌트로 활용된다.

Toast.tsx(4,601 bytes)는 시스템 알림 메시지를 표시하는 토스트(Toast) 컴포넌트를 구현한다. success, error, warning, info의 4가지 알림 유형에 따른 색상 및 아이콘 차별화, 자동 소멸 타이머(기본 3초), 수동 닫기 버튼, 스택 방식의 다중 알림 표시를 지원한다.

EventToast.tsx(5,664 bytes)는 WebSocket을 통해 수신되는 실시간 이벤트를 Toast 알림으로 변환하는 전용 컴포넌트이다. 이력서 처리 완료, 평가 완료, 리포트 생성 완료 등 비동기 작업의 완료 알림을 사용자에게 실시간으로 전달한다. EventBusContext를 구독하여 새로운 이벤트 발생 시 자동으로 Toast를 생성한다.

### 6.2.2 인증 컴포넌트 (components/auth/)

인증 컴포넌트는 사용자 인증 관련 UI 기능을 담당하는 3개의 모달 컴포넌트로 구성된다.

LoginModal.tsx(4,009 bytes)는 이메일/비밀번호 기반 로그인 폼을 제공한다. 입력 유효성 검증(이메일 형식, 비밀번호 최소 길이), 로그인 API 호출(/api/auth/login), JWT 토큰 수신 및 AuthContext 상태 갱신, 로그인 실패 시 에러 메시지 표시, 소셜 로그인(카카오, 구글, 네이버) 버튼이 포함된다.

RegisterModal.tsx(7,418 bytes)는 회원가입 폼을 제공한다. 이메일, 비밀번호, 비밀번호 확인, 이름 필드의 유효성 검증, 비밀번호 강도 표시기(Strength Indicator), 이용약관 동의 체크박스, 회원가입 API 호출(/api/auth/register) 및 자동 로그인 처리가 포함된다.

ForgotPasswordModal.tsx(4,273 bytes)는 비밀번호 재설정 기능을 제공한다. 등록된 이메일 주소 입력, 비밀번호 재설정 링크 발송 API 호출, 발송 완료/실패 상태 표시가 포함된다.

### 6.2.3 감정 분석 컴포넌트 (components/emotion/)

감정 분석 컴포넌트는 DeepFace 및 Hume Prosody 분석 결과를 실시간으로 시각화하는 전용 컴포넌트를 제공한다. 현재 지배적 감정의 아이콘 및 텍스트 표시, 7가지 감정의 확률 분포 바 차트, 감정 변화 추이 미니 그래프, 시선 상태(정면/이탈) 인디케이터 등의 UI 요소가 포함된다. WebSocket을 통해 1초 간격으로 수신되는 감정 분석 결과에 따라 컴포넌트 상태가 실시간으로 갱신된다.

### 6.2.4 리포트 컴포넌트 (components/report/)

리포트 컴포넌트는 면접 결과 리포트 페이지에서 사용되는 시각화 및 데이터 표시 컴포넌트를 제공한다. Recharts 및 Chart.js 라이브러리를 래핑한 차트 컴포넌트(RadarChart, BarChart, PieChart, AreaChart, RadialBarChart), 5축 평가 점수 카드, 합격/불합격 배지, STAR 분석 결과 표, 개선 권고사항 목록, PDF 다운로드 버튼 등의 컴포넌트가 포함된다.

## 6.3 상태 관리 (AuthContext, ToastContext, EventBusContext)

전역 상태 관리는 React Context API를 통해 구현되며, 외부 상태 관리 라이브러리(Redux, Zustand 등)에 대한 의존성 없이 경량 상태 관리를 달성한다. src/contexts/ 디렉토리에 3개의 Context가 정의되어 있으며, RootLayout(layout.tsx)에서 AuthProvider와 ToastProvider가 중첩 프로바이더(Nested Provider)로 래핑되어 하위 모든 컴포넌트에서 접근 가능하다.

### 6.3.1 AuthContext (4,677 bytes)

AuthContext는 사용자 인증 상태의 전역 관리를 담당한다. 관리되는 상태(State)로는 isAuthenticated(로그인 여부), user(사용자 정보 객체: 이메일, 이름, 역할 등), token(JWT 액세스 토큰), isLoading(인증 상태 확인 중 여부)이 포함된다.

제공되는 액션(Action)으로는 login(이메일/비밀번호 인증 후 토큰 저장), logout(토큰 제거 및 상태 초기화), register(회원가입 후 자동 로그인), socialLogin(OAuth2 소셜 로그인 처리), refreshUser(사용자 정보 갱신)가 정의된다.

JWT 토큰은 localStorage에 저장되며, 애플리케이션 초기 로드 시 저장된 토큰의 유효성을 검증하여 자동 로그인 상태를 복원한다. 모든 인증이 필요한 API 호출 시 Authorization: Bearer {token} 헤더가 자동으로 추가된다.

### 6.3.2 ToastContext (8,104 bytes)

ToastContext는 시스템 전역 알림 메시지의 생명주기를 관리한다. Toast 큐(Queue)를 통해 다중 알림의 순서적 표시를 지원하며, 각 알림은 고유 ID, 유형(success/error/warning/info), 메시지 텍스트, 자동 소멸 타이머(기본 3~5초, 유형별 차등), 수동 닫기 콜백으로 구성된다.

제공되는 API로는 showToast(message, type, duration)가 정의되며, 성공 알림은 녹색, 오류 알림은 빨간색, 경고 알림은 주황색, 정보 알림은 파란색의 시각적 구분이 적용된다. 알림은 화면 우상단에 스택(Stack) 방식으로 쌓이며, 최신 알림이 최상단에 표시된다. 알림 큐의 최대 크기가 설정되어 과도한 알림 축적으로 인한 UI 혼잡이 방지된다.

### 6.3.3 EventBusContext (5,455 bytes)

EventBusContext는 WebSocket 연결 상태 관리 및 실시간 이벤트 구독(Subscription) 시스템을 제공한다. FastAPI 서버의 WebSocket 엔드포인트(/ws/events)와의 연결을 유지하며, 서버에서 발행되는 30종 이상의 이벤트 타입을 수신하여 적절한 핸들러에 디스패치한다.

관리되는 상태로는 connectionStatus(연결 상태: connecting, connected, disconnected, error), lastEvent(가장 최근 수신된 이벤트 객체)가 포함된다. 재연결 로직(Reconnection Logic)이 구현되어, WebSocket 연결이 끊어진 경우 지수 백오프(Exponential Backoff) 전략으로 자동 재연결을 시도한다.

이벤트 핸들링 흐름은 다음과 같다. 서버에서 WebSocket 메시지가 수신되면 이벤트 타입(event_type)에 따라 등록된 핸들러가 호출된다. 이력서 처리 완료(resume.processed) 이벤트 수신 시 대시보드의 이력서 상태가 자동 갱신되고 Toast를 통해 "이력서 처리가 완료되었습니다" 알림이 표시된다. 평가 완료(evaluation.completed) 이벤트 수신 시 면접 세션 페이지의 진행 정보가 갱신된다. 리포트 생성 완료(report.generated) 이벤트 수신 시 리포트 페이지로의 이동 링크가 포함된 Toast가 표시된다.

## 6.4 실시간 이벤트 알림 시스템 (WebSocket + Toast)

실시간 이벤트 알림 시스템은 EventBusContext와 ToastContext의 협력적 운용을 통해 구현되며, 백엔드에서 발생하는 비동기 이벤트를 사용자에게 즉각적으로 전달하는 기능을 제공한다.

### 6.4.1 이벤트 수신 파이프라인

프론트엔드의 실시간 이벤트 수신 파이프라인은 3계층으로 구성된다.

제1계층은 WebSocket 전송 계층이다. EventBusContext가 FastAPI 서버의 WebSocket 엔드포인트와 상시 연결(Persistent Connection)을 유지하며, JSON 형식의 이벤트 페이로드를 수신한다.

제2계층은 이벤트 디스패처(Event Dispatcher) 계층이다. 수신된 이벤트의 event_type 필드를 기반으로 등록된 핸들러 함수를 호출한다. 핸들러는 useEffect 훅을 통해 각 페이지/컴포넌트에서 등록되며, 컴포넌트 언마운트(Unmount) 시 자동으로 구독이 해제된다.

제3계층은 UI 반영(UI Reflection) 계층이다. 핸들러 함수가 React 상태(State)를 갱신하거나 ToastContext의 showToast API를 호출하여, 수신된 이벤트를 시각적 피드백(차트 갱신, 텍스트 업데이트, 알림 표시 등)으로 변환한다.

### 6.4.2 주요 실시간 이벤트 유형

면접 세션 페이지에서 처리되는 실시간 이벤트는 다음과 같다.

STT 중간 결과(stt.interim) 이벤트는 지원자의 발화를 실시간으로 전사하여 화면에 표시하며, 말하는 중에도 텍스트가 점진적으로 업데이트되어 시스템이 음성을 인식하고 있음을 시각적으로 확인할 수 있다. STT 최종 결과(stt.final) 이벤트는 발화가 완료된 후 확정된 전사 결과를 채팅 이력에 추가한다. 감정 분석 결과(emotion.analyzed) 이벤트는 1초 간격으로 수신되어 감정 인디케이터 컴포넌트의 상태를 갱신한다. 질문 생성 완료(question.generated) 이벤트는 AI 면접관의 새로운 질문 텍스트와 TTS 오디오 URL을 전달하며, SSE 스트리밍과 병행하여 처리된다.

### 6.4.3 EventToast 통합 컴포넌트

EventToast.tsx 컴포넌트는 EventBusContext와 ToastContext를 연결하는 브릿지(Bridge) 역할을 수행한다. 이 컴포넌트는 RootLayout의 하위에 배치되어 애플리케이션 전역에서 동작하며, 특정 이벤트 타입(이력서 처리 완료, 리포트 생성 완료, 시스템 오류 등)을 자동으로 Toast 알림으로 변환한다. 이를 통해 각 페이지의 개별 컴포넌트가 직접 이벤트 핸들링 로직을 구현하지 않아도, 중요한 비동기 이벤트가 사용자에게 자동으로 알림된다.

## 6.5 반응형 디자인 및 접근성 (a11y)

### 6.5.1 반응형 레이아웃 설계

Tailwind CSS v4의 반응형 유틸리티 클래스(sm:, md:, lg:, xl:, 2xl:)를 활용하여, 모바일(~640px), 태블릿(641~1024px), 데스크톱(1025px~)의 3개 브레이크포인트에서 최적화된 레이아웃이 제공된다.

면접 세션 페이지에서는 데스크톱 환경에서 비디오 영역과 채팅 영역이 좌우 2열 레이아웃으로 배치되며, 모바일 환경에서는 비디오가 상단, 채팅이 하단에 배치되는 단일 열 레이아웃으로 전환된다. 코딩 테스트 페이지에서는 Monaco Editor의 높이가 뷰포트 크기에 따라 동적으로 조절되며, 문제 설명 패널과 코드 에디터 패널의 비율이 브레이크포인트에 따라 변경된다.

### 6.5.2 웹 접근성 (Web Accessibility, a11y)

웹 접근성 WCAG 2.1 가이드라인을 준수하기 위한 다음의 설계가 적용되었다.

시맨틱 HTML(Semantic HTML)의 적절한 활용이 이루어졌다. header, nav, main, section, article, footer 등의 시맨틱 태그를 통해 페이지 구조를 의미론적으로 마크업하여, 스크린 리더(Screen Reader)가 페이지 구조를 정확히 파악할 수 있도록 한다.

키보드 네비게이션(Keyboard Navigation)이 지원된다. 모든 인터랙티브 요소(버튼, 링크, 입력 필드, 모달)는 Tab 키를 통한 순차적 포커스 이동이 가능하며, 모달의 ESC 키 닫기, Enter 키 제출 등의 키보드 인터랙션이 구현되어 있다. 모달이 열릴 때 포커스 트랩(Focus Trap)이 적용되어, Tab 키가 모달 내부의 인터랙티브 요소들 사이에서만 순환하도록 제한된다.

색상 대비(Color Contrast)가 WCAG AA 수준(최소 대비율 4.5:1)을 충족하도록 Tailwind CSS의 색상 팔레트가 선정되었다. 특히 Toast 알림의 4가지 유형(success/error/warning/info)에서 배경색과 텍스트색의 대비가 충분히 확보되어, 색각 이상(Color Vision Deficiency) 사용자도 알림 유형을 시각적으로 구분할 수 있다.

ARIA(Accessible Rich Internet Applications) 속성이 적절히 사용된다. 모달의 role="dialog" 및 aria-modal="true", 버튼의 aria-label(아이콘 전용 버튼에 대한 대체 텍스트), 실시간 상태 변경 영역의 aria-live="polite"(스크린 리더가 상태 변화를 자동으로 읽어줌) 등이 적용되어 있다.

antialiased 폰트 렌더링(body className="antialiased")이 적용되어 모든 텍스트의 가독성이 향상되었다.

## 6.6 데이터 시각화 (Recharts + Chart.js)

### 6.6.1 Recharts 기반 시각화 (5종)

Recharts 라이브러리(≥ 3.7.0)는 React 네이티브 컴포넌트로 설계되어, 선언적(Declarative) API를 통해 다음 5종의 인터랙티브 차트를 구현한다.

RadarChart(레이더 차트)는 5축 언어 평가(문제해결력, 논리성, 기술이해도, STAR, 전달력)를 다각형 그래프로 시각화한다. 5개 축이 정오각형의 꼭짓점에 대응하며, 각 축의 점수가 중심에서 꼭짓점까지의 거리로 표현된다. CustomTooltip 컴포넌트를 통해 마우스 오버 시 해당 축의 정확한 점수와 등급이 표시된다.

BarChart(바 차트)는 각 질문별 평가 점수의 분포를 수직 막대 그래프로 표시한다. X축은 질문 번호(Q1~Q10), Y축은 총점(25점 만점)이며, 각 막대는 5개 축의 점수가 색상으로 구분된 누적 바(Stacked Bar)로 표현된다.

AreaChart(영역 차트)는 턴별 발화 속도(SPM) 변화를 시간축 그래프로 시각화한다. X축은 턴 번호, Y축은 SPM 값이며, 적정 범위(280~380 SPM)가 녹색 음영 영역으로 표시되어 지원자의 발화 패턴이 적정 범위 내에 있는지를 직관적으로 파악할 수 있다.

RadialBarChart(방사형 바 차트)는 시선 방향 분포(center, left, right, up, down, away)를 6개의 방사형 막대로 시각화한다. 각 방향의 비율이 방사 길이로 표현되며, center(정면 응시) 막대는 강조 색상으로 표시된다.

수평 BarChart(수평 바 차트)는 기술 키워드 출현 빈도를 수평 막대 그래프로 시각화한다. 상위 10개 기술 키워드가 빈도 내림차순으로 정렬되어 표시된다.

### 6.6.2 Chart.js 기반 시각화 (2종)

Chart.js(≥ 4.5.1)와 react-chartjs-2(≥ 5.3.1) 라이브러리는 Recharts가 직접 지원하지 않는 특수 차트 유형에 대한 보완적 시각화를 제공한다.

PieChart/DoughnutChart(파이/도넛 차트)는 DeepFace 기반 7가지 감정의 전체 세션 분포를 원형 차트로 시각화한다. 각 감정에 대응하는 색상이 할당되며(happy: 노란색, sad: 파란색, angry: 빨간색, neutral: 회색 등), 가장 높은 비율의 감정 영역에 레이블이 표시된다. Canvas API 기반 렌더링을 통해 대규모 데이터 포인트에서도 부드러운 애니메이션 전환이 유지된다.

STAR 구조 분석 차트는 Situation, Task, Action, Result 4개 요소의 키워드 출현 빈도를 가로 바 차트로 시각화한다. 각 요소의 점수(0~100)가 바의 길이로 표현되며, 요소별 색상 차별화를 통해 STAR 기법의 균형적 활용 여부를 시각적으로 평가할 수 있다.

### 6.6.3 차트 인터랙션 설계

모든 차트에는 다음의 인터랙션 기능이 공통적으로 적용된다.

Tooltip(툴팁)은 마우스 오버 시 해당 데이터 포인트의 정확한 수치, 레이블, 백분율 등의 상세 정보를 표시하는 팝업 박스이다. Recharts의 커스텀 Tooltip 컴포넌트를 통해 차트 유형에 맞는 포맷으로 정보가 표시된다.

Legend(범례)는 차트의 데이터 시리즈를 식별하기 위한 색상/기호 매핑 표로, 클릭 시 해당 데이터 시리즈의 표시/숨김을 토글할 수 있는 인터랙션을 제공한다.

반응형 크기 조절은 ResponsiveContainer 래퍼 컴포넌트를 통해 차트가 부모 컨테이너의 크기에 맞추어 자동으로 리사이즈되도록 구현되어, 다양한 화면 크기에서 최적의 시각화를 제공한다.

진입 애니메이션은 차트 데이터가 로드될 때 데이터 포인트가 순차적으로 나타나는 진입(Entry) 애니메이션이 적용되어, 시각적으로 매력적인(Visually Engaging) 사용자 경험을 제공한다.
